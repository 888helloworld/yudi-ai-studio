# 产品框架

这份文档回答：御弟哥哥到底给谁用、解决什么问题、为什么现在重点是画面工坊。

## 一句话定位

御弟哥哥是一个 AI 图文创作工作台：让用户用积分制完成文案、配图、参考图改图、看图写 Prompt，并让管理员能管理用户、积分、卡密和订单。

更白话一点：它是一个“会写、会画、会记账、还能让后台管起来”的小型创作系统。

## 目标用户

| 用户 | 想完成什么 | 现在的麻烦 | 产品机会 |
| --- | --- | --- | --- |
| 内容创作者 | 快速做小红书封面、配图、标题、正文 | 写一句改半天，图也难统一 | 从主题到成稿一步缩短 |
| 电商卖家 | 做商品主图、场景图、模特穿搭图 | 找摄影成本高，AI 图难复用 | 用参考图和 Prompt 批量试图 |
| 设计/运营团队 | 批量测试风格、沉淀提示词 | 每个人玩法不一样，结果难追踪 | 记录历史、复用图片和提示词 |
| 管理员 | 管账号、积分、卡密、订单 | 谁用了多少、失败退没退很难查 | 后台统一管理和追踪 |

## 当前核心模块

### 画面工坊

`image-studio.html` 是当前主角。

它负责：

- `gpt-image-2` 文生图
- 1-4 张参考图改图
- 看图写 Prompt
- 批量生成
- 并发控制
- 生成记录、预览、下载、重新生成
- 把历史图片继续作为参考图使用

当前页面只开放实测可用的方图规格，服务端会记录每张图的真实输出尺寸。`gpt-image-2` 官方约束支持任意合法 `WIDTHxHEIGHT`：宽高为 16 的倍数，比例在 1:3 到 3:1 之间，且不超过 3840x2160 等量像素；但当前接入的上游实际会忽略多种请求尺寸。

| 页面显示 | 请求上游尺寸 | 真实输出校验 |
| --- | --- | --- |
| 当前实测方图 | `1024x1024` | 当前实测 `1254x1254` |

2026-06-29 实测：`1024x1024`、`1024x1536`、`1536x1024`、`2048x1152`、`2560x1440` 都实际返回 `1254x1254`。页面不再展示竖图/横图，避免误导用户。

### 小红书工作台

面向小红书内容生产：

- 图片生成
- 标题生成
- 正文生成
- 标签生成
- 文案改写
- 图文一体

### 看图写 Prompt

用户上传图片后，系统反推出可复用提示词。

典型用法：

- 学习参考图构图和光线。
- 把一张商品图变成一段可复用 Prompt。
- 分别输出中文和英文提示词。
- 一键把提示词带回画面工坊继续生图。

### 后台和积分

后台要解决的不是“好看”，而是“管得住”：

- 用户管理
- 积分充值
- 积分流水
- 卡密和邀请码
- 历史记录
- 支付订单
- 系统配置检查
- 上游 ping

## 北极星指标

**7 日内成功创作并保存的有效内容数。**

有效内容定义：

- 图片：生成成功并进入历史记录。
- 改图：至少成功返回 1 张图片并进入历史记录。
- 看图写 Prompt：成功生成 Prompt 并进入历史记录。
- 文案：生成或改写成功并进入历史记录。
- 图文一体：图片或文案至少一项成功，且进入历史记录。

## AARRR 漏斗

| 阶段 | 目标 | 指标 |
| --- | --- | --- |
| Acquisition 获取 | 让用户注册或登录 | 页面访问、注册成功率、邀请码使用率 |
| Activation 激活 | 完成第一次有效创作 | 首创成功率、注册到首创耗时 |
| Retention 留存 | 用户持续回来创作 | D1/D7 留存、7 日创作次数 |
| Revenue 收入 | 用户购买、兑换或消耗积分 | 充值转化率、卡密兑换率、积分消耗 |
| Referral 推荐 | 用户传播邀请码或工具 | 邀请码生成数、邀请注册数 |

## MVP 边界

必须有：

- 登录注册和 JWT 鉴权。
- 本地/内网免邀请码，公网需要邀请码。
- 图片生成、参考图改图、看图写 Prompt。
- 文案生成、文案改写、图文一体。
- 积分扣减、失败退款、流水记录。
- 历史记录和真实输出尺寸记录。
- 管理员用户管理、充值、卡密、订单。
- 帮助、隐私、条款和内容规范入口。

暂不做：

- 自动发布到小红书。
- 多团队组织架构。
- 多模型自由切换市场。
- 正式支付自动入账。
- 复杂素材库和多人审批流。

## 体验原则

- 先让用户能生成，再解释复杂规则。
- 尺寸要按实际出图展示，不要拿请求参数糊弄用户。
- 失败必须给可行动反馈，扣了积分就要能退。
- 历史记录是素材库，不只是日志。
- 价格、积分、并发这些词要尽量大白话。
- 管理后台少装饰，多筛选、多状态、多排查信息。

## 缺口清单

| 缺口 | 优先级 | 说明 |
| --- | --- | --- |
| 正式支付验签 | P0 | 目前仍是模拟订单/人工确认 |
| `/uploads` 私有化 | P0 | 当前知道链接即可访问，隐私图风险高 |
| XSS 和 token 安全 | P0 | JWT 在 localStorage，后续建议 HttpOnly Cookie |
| 上游尺寸监控 | P0 | gpt-image-2 尺寸会变，需要持续记录真实输出 |
| 埋点和漏斗 | P1 | 当前更偏结果统计，过程数据不足 |
| 生成质量反馈 | P1 | 用户可标记好/差/重试原因 |
| 模板体系 | P1 | 高频场景需要 Prompt 模板沉淀 |
| 对象存储 | P1 | 高并发或生产环境不建议长期只用本地 uploads |
| 内容合规审核 | P2 | 降低违规、低俗、侵权内容风险 |
