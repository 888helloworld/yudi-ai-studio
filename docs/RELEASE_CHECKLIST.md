# 发布验收清单

每次准备发布前，按这份清单过一遍。它的目的不是好看，是防止上线后用户画不了图、积分扣错、数据丢了。

## 代码检查

- `npm run check` 通过。
- 没有把 `.env`、`data.db`、`uploads/`、`logs/`、`node_modules/` 加进提交。
- 没有把真实 API Key、JWT 密钥、管理员密码、支付密钥写进代码或文档。
- Git 提交只包含本次要发布的文件。
- README 和 `docs/` 的口径一致。

## 功能验收

- 登录、注册、退出流程正常。
- 本地/内网注册免邀请码符合预期。
- 公网注册需要邀请码符合预期。
- 画面工坊可以文生图。
- 画面工坊可以上传 1-4 张参考图改图。
- 看图写 Prompt 可以返回中文和英文提示词。
- 小红书图片生成、文案生成、文案改写、图文一体均可用。
- 历史记录可查看、分页、删除。
- 生成结果可预览、下载、重新生成。
- 历史图片可复用为参考图。
- 管理员可查看用户、充值、重置密码、生成卡密、查看订单。

## 画面工坊尺寸验收

页面显示当前 4 个常用规格。服务端按 `gpt-image-2` 官方约束放行合法显式尺寸，并记录真实输出尺寸。

- 1:1 方图：实际 `1254x1254`，请求 `1024x1024`
- 2:3 竖图：实际 `1024x1536`，请求 `1024x1536`
- 3:2 横图：实际 `1536x1024`，请求 `1536x1024`
- 16:9 横图：实际 `1672x941`，请求 `2560x1440`

必须确认：

- 4K 按钮不出现。
- 页面显示的是实际出图尺寸，不只显示请求尺寸。
- 尺寸按钮不换行乱跳。
- 尺寸不匹配时仍回传图片。
- 历史记录里能看到输出尺寸。
- `XI_XU_NORMALIZE_OUTPUT_SIZE=false` 时不强制改图画布。

## 积分和退款验收

- 图片成功生成后扣 `10 积分/张`。
- 文案生成扣 5 积分。
- 文案改写扣 3 积分。
- 看图写 Prompt 扣 5 积分。
- 图文一体按文案和图片组合扣费。
- 积分不足时不能创建任务。
- 上游失败时自动退款。
- 少出图时按缺少张数退款。
- 服务重启后未完成任务会失败并退款。
- 积分流水能追溯扣费和退款原因。

## 队列和并发验收

- 前端“要生成几张”含义清楚。
- 前端“同时生成几张”含义清楚。
- 服务端 `XI_XU_MAX_ACTIVE_JOBS=0` 时表示不限制服务端任务并发。
- 图片接口限流生效：`XI_XU_IMAGE_RATE_LIMIT_PER_MIN`。
- 排队、运行、完成、失败数量显示合理。
- 刷新页面后仍能看到历史记录。

## 安全验收

- 只暴露需要的静态页面和上传资源。
- `.env`、`data.db`、源码、日志、测试文件无法通过浏览器下载。
- 上传文件做 MIME 和文件头双重校验。
- 上传大小限制生效。
- 生成接口必须登录。
- 管理员接口普通用户无法访问。
- 图片接口有速率限制和积分校验。
- 前端渲染用户输入和模型输出时不使用不可信 `innerHTML`。
- JWT 当前在 localStorage，生产环境风险已记录；后续计划迁移 HttpOnly Cookie。
- `/uploads` 当前不是严格私有资源，生产隐私场景必须升级鉴权或签名 URL。

## 支付验收

当前支付仍是模拟订单/人工确认：

- `/api/payment/create` 可以创建订单。
- 管理后台可以手动确认到账。
- 模拟支付回调默认关闭。
- 如开启模拟回调，必须配置 `MOCK_PAYMENT_TOKEN`。
- 真实支付宝/微信回调未完成验签前不能自动入账。

正式收费前必须确认：

- 平台签名校验完成。
- 金额二次校验完成。
- 订单幂等完成。
- 重复回调不会重复加积分。
- 对账流程明确。
